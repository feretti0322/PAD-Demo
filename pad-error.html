🛡️ エラーハンドリングは「失敗時の備え」

700名分の処理途中で1名だけページが壊れていた場合、そこで全体を止めると残り699名分が無駄になります。「止めるべきエラー」と「止めずに続けるべきエラー」の使い分けこそが、PADを業務に乗せる人と趣味で終わる人を分ける決定的なスキルです。

① ON BLOCK ERROR の基本

PADでは各アクションに対して「On block error」を設定できます。エラー発生時の挙動を以下から選びます。

  • 新しいルールを追加 → サブフローへ：エラー時に独自のリカバリ処理を呼び出す
  • エラー発生時の動作 → フローを停止：そこで全体中断（既定）
  • エラー発生時の動作 → 次の繰り返しに移動：ループ内で個別スキップ
  • エラー発生時の動作 → ブロックの末尾へ：エラーをなかったことにして続行

② 「全体中断 vs 個別スキップ」の使い分け（最重要）

場面設定理由
ログインに失敗した場合 フロー全体を中断 ログインできなければ以降の全処理が無意味になるため
700名処理中に1名だけページが取れなかった 次の繰り返しに移る 残り699名分の処理を続けるべき。その1名はログに記録して後で確認
Excel保存時のファイル占有エラー リトライ後にスキップ 誰かが開いている可能性。ログに残して次へ
設計判断のキモ： 「このエラーが起きたら、続けても意味があるか？」を毎回問う。意味があれば 個別スキップ、意味がなければ 全体中断。

③ リトライ（再試行）パターン

ネットワーク遅延などで一時的に失敗するケースは、3回までリトライしてから諦めるのが定番です。

retry = 0 LOOP WHILE retry < 3 ON BLOCK ERROR (次のアクションへ移動) ページに移動: %url% EXIT LOOP # 成功したらループ脱出 END retry = retry + 1 Wait: 2 seconds END IF retry = 3 THEN THROW "ページ取得に3回失敗"

④ エラーログの記録

スキップしたスタッフのIDをテキストファイルに追記しておくと、後追い確認が楽になります。

FOR EACH staff IN StaffList ON BLOCK ERROR (次の繰り返しに移動) # 詳細ページ取得 → Excel出力 Navigate: staff-detail.html?id=%staff['ID']% ... END # ON BLOCK ERROR の末尾でログを書く File append: %errorLogPath% text: "%LoopIndex%: %staff['ID']% スキップ" END

⑤ インタラクティブデモ：エラー動作シミュレーター

5名分のスタッフを処理します。3件目（ST003）は必ずエラーを起こす設定です。「全体中断モード」と「個別スキップモード」で結果がどう変わるか見てみましょう。

📝 3行まとめ

  1. ON BLOCK ERROR でエラー時の挙動を制御できる。
  2. 「ログインの失敗 → 全体中断」「個別データの欠損 → スキップ」の使い分けが業務適用の鍵。
  3. スキップした件はログに残す。「動いた／動かなかった」だけでなく「何件成功・何件失敗」を可視化する。
